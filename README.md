# astrbot_plugin_group_member_context

让 AstrBot 在群聊里自动感知并使用群成员身份信息。

## 作用

这个插件会在群聊消息进入 LLM 之前，默认只整理并注入**当前发言者**的信息：

- 当前发言者的 QQ
- 当前发言者的昵称
- 当前发言者的群昵称 / 群名片
- 当前发言者的专属头衔
- 当前发言者的群身份（群主 / 管理员 / 普通成员）
- 当前发言者的群等级

对于**其他成员**的信息，例如：

- 当前群的群主是谁
- 当前群有哪些管理员
- 另一个 QQ 是谁
- 某个人是不是管理员
- 某个人的群名片、头衔、身份是什么

则交给 LLM Tool 按需调用查询。

这样模型在平时对话时，至少能知道“当前说话的人是谁”；需要查其他成员时，再走工具查询，不会每轮都扫全群。

## 实现方式

插件使用 OneBot 11 / aiocqhttp 接口：

- `get_group_member_info`
- `get_group_member_list`

插件支持两种工作模式：

1. **自动注入模式**
   - 在 `@filter.on_llm_request()` 阶段把当前发言者的真实身份信息注入到系统提示词中
   - 优先直接使用群消息原始事件里的 `sender` 字段
   - 只有事件里拿不到时，才回退到 `get_group_member_info`

2. **LLM 工具模式**
   - 注册工具，让模型自己决定要不要调用
   - 适合查询其他群主、其他管理员、其他群成员信息

当前提供两个 LLM 工具：

- `get_group_identity_snapshot`
  - 获取当前群的身份快照，适合查询群主、管理员、Bot 身份等整群信息
- `query_group_member_identity`
  - 按关键词查询群成员身份信息

## 适用环境

- AstrBot
- QQ / OneBot 11 / aiocqhttp 适配器
- 机器人具有正常获取群成员信息的能力

## 安装方法

把本目录放进 AstrBot 插件目录中，目录名保持为：

`astrbot_plugin_group_member_context`

然后在 AstrBot 插件管理中启用它。

## 配置项

### `enable_auto_inject`
- 类型：布尔
- 默认：`true`

是否自动注入当前发言者的身份上下文。

- 开启：每轮在进入 LLM 前自动补充“当前发言者”身份信息
- 关闭：不自动注入，只注册 LLM 工具，让模型自己决定是否调用

如果你要“平时只靠 llmtool，不自动注入”，把这个配置改成 `false` 即可。

### `inject_group_admin_list`
- 类型：布尔
- 默认：`true`

是否在工具查询结果中附带管理员列表。

### `smart_query_group_snapshot`
- 类型：布尔
- 默认：`true`

保留该配置用于群身份快照查询相关逻辑。  
在当前设计下，日常自动注入不会依赖整群快照；整群信息更适合由工具按需查询。

### `sender_info_ttl_seconds`
- 类型：整数
- 默认：`120`

当前发言者信息缓存秒数。  
不过当前实现会**优先使用原始事件里的 sender 信息**，只有拿不到时才回退到 `get_group_member_info`，所以日常开销通常更低。

### `group_snapshot_ttl_seconds`
- 类型：整数
- 默认：`300`

群身份快照缓存秒数。  
用于缓存：

- 群主
- 管理员列表
- Bot 在当前群里的身份

在缓存有效期内，不会重复调用 `get_group_member_list`。

### `no_cache`
- 类型：布尔
- 默认：`false`

是否要求 OneBot 接口本身也尽量不使用缓存。  
通常建议保持关闭，让插件自己的内存缓存生效即可。

### `extra_instruction`
- 类型：文本
- 默认：空

会附加到插件自动注入的提示词后面，可用于追加你的个性化规则。

## 效果说明

例如群里有一条原始事件：

```text
sender: {
  user_id: 2595044177,
  nickname: "穆斯塔菲",
  card: "海绵宝宝哈耶克",
  role: "admin",
  level: "28",
  title: "传奇与无敌"
}
```

插件会优先把这些信息整理后注入给模型，让模型知道：

- 这句话是谁发的
- 对方 QQ 是多少
- 对方昵称是什么
- 对方群名片是什么
- 对方是不是管理员 / 群主 / 普通成员
- 对方头衔是什么
- 对方群等级是什么

也就是说，如果群里有人发：

> 你们都注意点

模型在理解这句话时，就能知道这句话是**某个管理员**说的，而不是把发言者当成普通用户。

如果后续用户再问：

- 谁是群主
- 谁是管理员
- 这个 QQ 是谁
- 某某是不是管理
- 某某的头衔是什么

则由 LLM Tool 再去查询其他成员信息。

## 推荐用法

### 方案一：发送者信息自动注入
适合你希望模型默认知道“当前发言者是谁”。

推荐配置：

```json
{
  "enable_auto_inject": true
}
```

### 方案二：纯 LLM 工具模式
适合你希望模型自由决定要不要查任何身份信息。

推荐配置：

```json
{
  "enable_auto_inject": false
}
```

这样插件仍然会注册：

- `get_group_identity_snapshot`
- `query_group_member_identity`

但不会每轮自动注入上下文，是否查询由模型自己决定。

### 推荐思路

最符合你这次需求的用法是：

```json
{
  "enable_auto_inject": true
}
```

即：

- 平时只注入当前发送者信息
- 查询其他群主 / 其他管理 / 其他群成员时，让模型调用工具

## 资源消耗说明

不是每次 LLM 请求都会完整查询一遍群主和管理员信息。

当前策略是：

1. **发言者信息优先直接取消息事件**
   - 优先使用群消息原始事件里的 `sender` 字段
   - 只有拿不到时才回退到 `get_group_member_info`
   - 因此日常开销更低

2. **其他成员 / 群主 / 管理员按需查**
   - 通过 LLM Tool 调用
   - 需要时才会查询 `get_group_member_list`
   - 并且默认缓存 `300` 秒

也就是说，像普通群聊消息：

- 今天天气
- 讲个笑话
- 帮我写代码
- 你们都注意点

通常只会读取当前事件里的发送者信息，**不会去扫整群成员列表**。

只有模型确实需要知道：

- 谁是群主
- 谁是管理员
- 某个 QQ 是谁
- 某个人是不是管理

才更适合调用工具查询。

所以资源消耗已经比“每轮都扫全群成员”小很多。

## 注意事项

1. 这个插件只增强“模型知道群成员身份信息”的能力，不会修改 AstrBot 原本的会话存储结构。
2. 如果你的平台不是 `AiocqhttpMessageEvent`，插件会自动跳过，不会报错中断。
3. 如果接口调用失败，插件会记录 warning 日志，并回退为不注入。
4. 如果你希望最省资源，建议：
   - 保持 `enable_auto_inject = true`
   - 保持 `no_cache = false`
   - 适当提高 `group_snapshot_ttl_seconds`

## 你这次需求对应的解决点

你给出的日志里，原始事件里其实已经有：

- `sender.role`
- `sender.nickname`
- `sender.card`
- `sender.title`
- `sender.user_id`
- `sender.level`

但在 AstrBot 的后续会话处理中，这些信息没有稳定传递到模型侧，所以模型才会回答错误。

这个插件现在的目标是：

1. **LLM 请求前优先从原始群消息事件里提取并整理发送者信息注入给模型**
2. **避免依赖框架中间层已经丢失的 `role=user` 结果**
3. **其他成员信息交给工具按需查询**

这正对应你说的：

- 平时只给 LLM 插入发送者的信息
- 查询其他群主 / 其他管理 / 其他群成员时才调用 llmtool
